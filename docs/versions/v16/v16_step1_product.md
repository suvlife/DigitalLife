# V16: 私聊消息排队与占位提交 - 产品文档

## 目标

在 V15 已提供“当前谁在工作”的基础反馈之后，V16 继续优化**私聊房间**的消息体验，使 Agent 在处理当前房间任务时，聊天窗口内能够直接出现输入中占位，并在完成后原地变为最终消息。

此前即使界面已经能看到 Agent 正在工作，消息流本身仍然是“要么没有消息，要么直接出现最终消息”。V16 的目标，是让私聊房间的消息演进更接近真实聊天软件：先看到占位，再看到结果，并且整条消息在视觉上保持连续。

---

## 功能特性

### 一、私聊房间输入中可见

- **消息流占位**：当 Agent 正在处理私聊房间发言任务时，消息流中会出现明确的“正在输入”占位消息。
- **更贴近聊天体验**：用户不只知道 Agent 正在工作，还能直接在消息流里看到这条回复正在形成。
- **仅私聊生效**：该能力第一版只针对 `PRIVATE` 房间启用，`GROUP` 房间继续保持直接插入消息。

### 二、原地变为最终消息

- **原地提交**：Agent 完成本轮发言后，占位消息直接变为最终消息，而不是在底部再新增一条。
- **位置稳定**：占位出现的位置，就是最终消息显示的位置。
- **阅读顺畅**：避免界面闪动、重复消息或用户把过程误解成两条独立消息。

### 三、消息顺序稳定

- **时序自然**：私聊房间在 Agent 输入期间，后续消息仍保持自然顺序，不因占位机制出现明显乱序。
- **结果可预期**：用户不会看到“输入中在上方，最终消息跳到底部”的情况。
- **视觉一致**：最终消息与中间占位之间保持一一对应关系。

### 四、异常状态可感知

- **失败可见**：若 Agent 在处理中失败，原占位消息会更新为明确的失败说明。
- **取消可结束**：若当前轮被取消或未产生有效发言，占位消息也会被正确收尾，不继续停留在“正在输入”。
- **状态可信**：私聊消息流中的占位状态应始终与真实运行状态一致。

### 五、群聊保持现状

- **群聊不引入消息排队**：`GROUP` 房间继续使用当前直接插入消息的方式，不在消息流里插入输入中占位。
- **避免多 Agent 语义复杂化**：群聊中不引入“已入库但暂不可读”的额外状态，保持 Agent 对消息可见性的理解简单一致。
- **与 V15 配合**：群聊需要的“谁正在工作”能力由 V15 提供，V16 不再重复处理。

---

## 用户价值

### 1. 私聊对话更自然

用户在与单个 Agent 交流时，能像使用即时通讯软件一样感知回复的生成过程。

### 2. 消息阅读更稳定

最终回复在原位置出现，不会因为占位结束而跳动或重复插入，降低阅读负担。

### 3. 异常反馈更明确

即使 Agent 处理失败，用户也能在原消息位置看到明确结果，而不是只留下一个消失或卡住的占位。

---

## 核心概念

### 占位消息

私聊房间消息流中的临时消息，用于表达“该 Agent 正在准备当前轮回复”。

### 原地提交

当 Agent 完成发言后，原有占位消息直接转变为最终消息，而不是新增另一条独立消息。

### 私聊消息排队

在私聊房间里，消息可见性会围绕当前输入中的消息进行有序推进，以保证展示顺序稳定。

---

## 产品边界

### V16 包含

- 私聊房间中的输入中占位消息
- 占位消息原地变为最终消息
- 失败、取消、未发言时的占位收尾
- 私聊消息顺序稳定与重启后状态恢复

### V16 不包含

- 群聊房间中的消息排队与占位提交
- 逐字流式输出动画
- 已读、送达、回执等复杂消息状态体系
- 历史消息的回放式输入动画

---

## 验收标准

### 私聊房间体验

- 当 Agent 正在处理私聊房间发言任务时，消息流中能看到明确的输入中占位
- Agent 完成本轮发言后，占位消息会在原位置变为最终消息
- 私聊房间中相关消息的展示顺序保持稳定，不出现明显跳位

### 异常处理

- Agent 失败时，原占位消息会更新为明确的失败说明
- Agent 被取消或本轮未发言时，占位消息能够被正确结束

### 范围控制

- `GROUP` 房间继续保持直接插入逻辑，不进入消息排队机制
