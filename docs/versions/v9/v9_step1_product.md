# V9: 数据持久化与重启恢复 - 产品文档

## 目标

在 V8 Agent 执行能力的基础上，为系统补齐运行状态持久化能力。服务不再是“一次性会话容器”，而是具备基础状态记忆的长期运行系统。即使后端进程重启，仍能够恢复之前的聊天室上下文、调度相关状态和 Agent 状态，为后续继续协作提供基础。

---

## 功能特性

- **消息历史持久化**：聊天室消息不再只保存在内存中，服务重启后仍可查看此前对话内容。
- **房间状态恢复**：恢复房间当前的调度状态，包括是否仍在运行、已进行到第几轮、当前轮到谁发言。
- **Agent 上下文延续**：恢复 Agent 的私有上下文，使其在重启后仍保留之前的对话记忆与协作背景。
- **恢复后手动续跑**：若服务停止时某些房间仍处于活跃状态，重启后先恢复其历史与调度位置，但是否继续运行由用户显式触发。
- **持久化可配置**：支持通过配置开启或关闭持久化能力，并指定本地存储位置，兼顾演示模式与长期运行模式。

---

## 用户价值

### 1. 让系统从“临时演示”变成“可持续运行”

此前版本中，服务一旦关闭，所有运行态都会丢失。V9 之后，系统可被用于更长时间的多 Agent 协作任务，而不必担心重启后完全清空上下文。

### 2. 降低运维成本

在开发、调试或部署过程中，服务重启是常见操作。V9 可以避免因升级、异常退出或手动重启而丢失全部现场。

### 3. 提升协作可信度

对于执行型 Agent 而言，长期上下文尤为重要。若每次重启后都“失忆”，会影响多轮协作的稳定性。V9 使 Agent 更接近真实的持续工作成员。

---

## 效果演示

### 场景示例

一个团队正在 `release-room@engineering` 中讨论版本发布：

```
--- release-room@engineering ---
PM:          本周五发布 V1.4，先确认风险项。
developer:   我会检查后端接口变更。
researcher:  我补一下外部依赖升级说明。
```

此时服务异常退出。

重启后，系统恢复此前状态：

```
--- release-room@engineering ---
[系统恢复] 已加载历史消息 38 条，当前房间恢复为 scheduling 状态。
[系统恢复] 当前应由 researcher@engineering 继续发言。
[系统恢复] 当前不会自动恢复调度，请由用户手动触发继续运行。
```

用户不需要重新建房、重新拉起团队，也不需要让 Agent 从头再聊一遍；但在 V9 当前阶段，恢复完成后是否继续调度，需要由用户手动触发。

## 产品边界

### V9 包含

- 恢复聊天室历史消息
- 恢复房间轮次与调度进度
- 恢复 Agent 上下文记忆
- 支持服务重启后从已恢复状态手动继续运行

### V9 不包含

- 多机共享状态
- 分布式高可用
- 多实例并发写入同一运行状态
- 跨版本历史状态自动迁移
- 服务重启后自动恢复聊天室调度

V9 的目标是先完成**单机单实例**下的稳定持久化与恢复。

---

## 验收标准

- [ ] 服务重启后，已存在房间的历史消息仍可查询与展示。
- [ ] 服务重启后，处于运行中的房间能够恢复到正确的调度位置，而不是从头开始。
- [ ] 服务重启后默认不自动恢复聊天室调度，需由用户手动重新触发。
- [ ] Agent 在恢复后仍能延续之前的对话上下文，不表现为完全失忆。
- [ ] 若房间在重启前已停止调度，重启后不应被错误地重新激活。
- [ ] 支持通过配置项启用或关闭持久化能力。
- [ ] 持久化异常时，系统应明确报错或降级，而不是静默丢失状态。

## 使用说明

### 配置方式

后端配置新增持久化开关和存储路径，例如：

```json
{
  "persistence": {
    "enabled": true,
    "db_path": "./runtime/state/teamagent.db"
  }
}
```

### 运行方式

启用后，系统启动时会自动：

1. 加载配置定义的 Team、Agent、Room
2. 检查本地状态库是否存在历史运行数据
3. 若存在，则恢复消息、房间状态和 Agent 上下文
4. 若不存在，则按首次启动逻辑初始化
5. 恢复完成后保持房间当前状态，但不自动恢复调度

### 观察方式

- 在日志中可看到“首次启动”或“恢复启动”的判定结果
- 在 Web API / TUI 中可继续查看重启前的消息历史
- 对处于活跃状态的房间，系统会显示其恢复出的调度位置，但不会自动续跑后续轮次
