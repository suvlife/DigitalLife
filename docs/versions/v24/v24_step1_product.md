# V24: 模型配置体系重构 - 产品文档

## 目标

重构 LLM 模型配置体系，解决当前"服务与模型耦合"的问题，并支持：

1. **预置提供商**：内置常见提供商（OpenAI、Anthropic、阿里云、DeepSeek 等），自动填充默认 URL，用户只需填 API Key。
2. **多协议 URL**：一个提供商可包含多个 URL，对应不同的 API 协议（OpenAI 兼容、Anthropic 等）。
3. **服务内多模型**：每个提供商下可配置多个模型，每个模型有独立属性。
4. **全局默认模型槽位**：取消原有的默认服务，改为主模型、轻量级模型、读图模型等语义化槽位。
5. **Agent 灵活绑定**：Agent 可绑定槽位或指定具体模型（`模型@提供商`）。

---

## 核心概念

### 1. 提供商（Provider）

一个提供商对应一个 API 服务提供商。系统将内置多种常见提供商类型（如 OpenAI, Anthropic, 阿里云, DeepSeek 等）。
每种内置提供商会自动配置对应的 API URL（包括不同协议的 URL）。用户可覆盖这些预置的 URL。

### 2. 模型（Model）

每个模型归属于某个提供商，拥有独立的属性配置，例如：
- 模型名称、显示名称。
- 是否支持多模态（读图）。
- 独立的上下文配置（包含上下文窗口、输出预留空间及压缩策略，可覆盖全局默认配置）。
- 指定使用的 API 协议（如指定使用 OpenAI 或 Anthropic 协议）。

### 3. 全局默认模型槽位

系统在全局层面定义了三个角色化的模型槽位，方便在不同场景中调度最合适的模型：
- **主模型 (Primary)**：用于复杂和主要的推理任务。
- **轻量级模型 (Lightweight)**：用于简单任务（如对话摘要、意图分类等）。
- **读图模型 (Vision)**：专门用于多模态图片识别场景。

### 4. 全局上下文与压缩策略

系统提供统一的全局 `context_config` 配置，用于集中管理所有的**上下文参数**（包含上下文窗口预算、输出预留空间，以及压缩触发阈值、摘要保留长度等压缩策略）。
> **覆盖机制**：为了满足不同模型的物理限制与定制需求，每个模型配置中也可以单独挂载一个 `context_config` 对象。当模型拥有自身的该配置时，将优先使用模型自身的设置覆盖全局默认配置。

### 5. Agent 模型绑定

Agent 在配置模型时支持以下几种方式：
- **使用系统主模型**：如果不配置，默认使用全局的主模型槽位。
- **引用默认槽位**：指定使用"主模型"、"轻量级模型"或"读图模型"。
- **直接指定模型**：明确指定使用某提供商下的特定模型（例如 `gpt-4o@openai-main`）。

> [!IMPORTANT]
> 为了保证 `模型@提供商` 格式的准确解析，**模型名称**和**提供商名称**中均不可包含 `@` 字符。

## 功能特性

### 一、预置提供商与多协议支持

- **自动填充**：新建提供商时选择类型（如"阿里云"），系统自动填充相关的预置 URL。
- **多协议 URL**：每个提供商可配置多个 URL，分别对应 OpenAI 兼容协议或 Anthropic 协议等，系统在调用时自动根据模型支持情况选择。

### 二、服务内多模型管理

- **多模型配置**：用户可以在同一个提供商下添加和管理多个模型。
- **独立属性**：可单独启用/禁用某个模型，并为其设置多模态能力。

### 三、全局默认模型槽位配置

- 用户可以在全局设置中，从现有的提供商模型列表中挑选出三个槽位（主模型、轻量级模型、读图模型）对应的具体模型。
- 读图模型槽位仅允许选择被标记为"多模态"的模型。

### 四、向后兼容与平滑迁移

- **配置版本化**：引入配置文件版本号机制，新结构被定义为 v2，而系统中现有未标注版本号的旧版配置均被视为 v1。
- **自动平滑迁移**：系统启动时检测到 v1 版本配置后，会自动将其无缝迁移为基于提供商与模型组合的新结构，并将旧有的默认模型自动映射到新的"主模型"槽位，无需用户手动干预。未来的架构演进也将基于此版本号机制进行平滑升级。

### 五、模型连通性测试

- **独立测试能力**：支持对单一模型进行独立的连通性测试，验证 API Token 和对应模型的可用性。
- **状态反馈**：用户可以直接在界面上触发测试，并实时查看连接结果与错误原因。

## 前端交互

### 提供商列表与模型管理

Web Console 设置页面的大模型配置区将被改造：
1. 顶部提供**默认模型槽位**的配置区域。
2. 下方以卡片形式展示每个**提供商**。
3. 每个提供商卡片内，列出其下属的**模型列表**，并提供针对每个模型的编辑与删除操作。
4. 新增提供商和编辑模型时，通过弹窗表单进行详细参数设置（如协议选择、Token 配置等）。

## 产品边界

### V24 包含

- 预置提供商类型及默认 URL。
- 提供商多协议 URL 配置及模型列表管理。
- 模型属性配置（名称、多模态、token 预算、协议配置）。
- 模型级别的独立连通性测试。
- 全局默认模型槽位管理。
- Agent 模型灵活绑定支持。
- 旧版配置格式自动迁移。
- Web Console 相关界面的适配重构。

### V24 不包含

- 自定义模型槽位（目前固定为三个语义化槽位）。
- API Key 加密存储（当前阶段明文）。
- 模型调用负载均衡机制。
- 按不同复杂任务场景自动动态路由模型。

## 验收标准

- [ ] 新建提供商时选择内置类型能自动填充正确的预置 URL。
- [ ] 一个提供商下可配置多个模型，各个模型可配置独立的名称、多模态标记及启用状态。
- [ ] 全局默认模型槽位（主模型、轻量级模型、读图模型）可被配置并在 Agent 中正常引用。
- [ ] 读图模型槽位下拉选项仅显示已启用的多模态模型。
- [ ] Agent 可通过明确指向 (`模型@提供商`) 指定所用模型，且未指定时正确降级为主模型槽位。
- [ ] 支持在界面上对具体模型进行独立的连通性测试，并能展示测试结果。
- [ ] 旧格式配置文件在系统启动时可成功自动迁移为新的提供商结构。
- [ ] Web Console 能够正确展示层级化的提供商及模型管理界面，并支持 CRUD 操作。

## 使用说明

### 新增提供商及模型

1. 进入 Web Console → 系统设置 → 大模型提供商管理。
2. 点击"新增提供商"，选择提供商类型，系统自动填充相关预置 URL。
3. 填入 API Key 后进入下一步，在提供商下添加具体的可用模型（如 gpt-4o、qwen-plus 等），可按需勾选"多模态"属性。

### 设置默认模型槽位

1. 在管理页面顶部"默认模型槽位"区域。
2. 根据下拉列表，为主模型、轻量级模型和读图模型指定具体的模型。
3. 设置立即生效，所有引用了槽位的 Agent 均会更新所用模型。

### 配置 Agent

在团队配置或 Agent 编辑页面中，可以为 Agent 指定模型策略：
- 选择对应的角色默认槽位（推荐）。
- 或者明确指定一个确切的具体模型（不受全局槽位更改的影响）。
