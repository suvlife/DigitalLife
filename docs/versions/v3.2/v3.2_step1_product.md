# V3.2: 以 Agent 为中心的事件驱动调度 - 产品文档

## 目标

重构调度模型，将 V3.1 中以**房间**为入口的调度改为以 **Agent** 为入口的事件驱动调度。每个 Agent 维护自己的事件队列，当有新消息到达时，事件被推入对应 Agent 的队列；调度器以 Agent 为单位并发运行，每个 Agent 独立消费自己的事件。这一模型更贴近现实中人类在多个群聊之间处理消息的方式。

---

## 与 V3.1 的核心区别

V3.1 中调度器是房间视角：逐房间推进轮次，由调度器决定"该哪个房间的哪个 Agent 发言"。

V3.2 中调度器是 Agent 视角：每个 Agent 独立运行，主动消费自己队列中的待处理事件；调度器只负责启动各 Agent 的事件循环，不再干预发言顺序。

| 维度 | V3.1 | V3.2 |
|------|------|------|
| 调度入口 | 房间（逐房间推进轮次） | Agent（每个 Agent 独立消费事件） |
| 发言触发 | 调度器按轮次调用 Agent | Agent 检测到新事件后自行发言 |
| 并发粒度 | 同一大轮内各房间并发 | 各 Agent 全程并发，互不干扰 |
| 跨房间感知 | 支持（同 V3.1） | 支持（同 V3.1） |
| 消息路由 | 同 V3.1 | 同 V3.1 |

---

## 功能特性

- 每个 Agent 持有一个事件队列，事件类型包括"收到房间消息"
- 同一房间内，参与者按配置顺序顺序发言：前一个参与者回复后，房间才将事件推送给下一位参与者，实现房间内的轮流发言效果
- 每个 Agent 的事件循环独立并发运行，各房间对话互不干扰
- 同时参与多个房间的 Agent（如 bob）串行消费自己的事件队列，不同房间的事件按到达顺序依次处理
- 跨房间上下文感知能力保留（同 V3.1）

## 效果演示

### 配置示例

同 V3.1：general 房间（alice、bob、charlie）+ tech 房间（bob、charlie、dave）

### 行为对比

**V3.1 行为**（调度器控制）：
```
大轮 1：调度器调用 general/alice，再调用 tech/bob
大轮 2：调度器调用 general/bob，再调用 tech/charlie
...
```

**V3.2 行为**（Agent 自驱动，房间内顺序推进）：
```
系统启动 → 所有 Agent 并发启动各自的事件循环

general 房间初始化 → 推送事件给 alice（第一位）
tech    房间初始化 → 推送事件给 alice（第一位）

general 房间：
  alice 收到事件 → 发言 → general 推送事件给 bob
  bob   收到事件 → 发言 → general 推送事件给 charlie
  charlie 收到事件 → 发言 → general 推送事件给 alice（下一轮）
  ...

tech 房间（与 general 并发推进）：
  alice 收到事件 → 发言 → tech 推送事件给 bob
  bob   收到事件 → 发言 → tech 推送事件给 charlie
  charlie 收到事件 → 发言 → tech 推送事件给 alice（下一轮）
  ...

bob 同时在两个房间，其事件队列可能积累来自两个房间的事件，
按到达顺序串行处理，不会同时在两个房间发言。
```

Bob 和 Charlie 同时活跃于两个房间，各房间独立推进轮次，bob 的跨房间发言顺序由事件到达顺序自然决定。

## 验收标准

- [ ] 每个 Agent 持有独立的事件队列
- [ ] 聊天室新消息自动推送事件到相关 Agent 的队列
- [ ] 所有 Agent 事件循环并发运行，互不阻塞
- [ ] Agent 按事件队列顺序处理消息并发言
- [ ] 各房间对话完整推进，消息不丢失、不重复
- [ ] 跨房间上下文感知能力保留（同 V3.1）
- [ ] 程序能正常退出（所有 Agent 事件队列耗尽后退出）

## 使用说明

### 配置文件

配置结构与 V3.1 保持一致，无需新增字段。

```json
{
  "cross_room_context": true,
  "agents": [...],
  "chat_rooms": [...]
}
```

### 运行

```bash
python src/main.py
```

### 输出

日志格式与 V3.1 一致，每条消息标注所属房间，日志保存到 `logs/` 目录。
