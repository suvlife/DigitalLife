# V3.1: 跨房间上下文感知 - 产品文档

## 目标

在 V3 多房间基础上，打破房间之间的上下文隔离：Agent 在发言时不仅能看到当前房间的消息历史，还能感知自己所在的其他房间的存在和对话内容。这使得同时活跃于多个房间的 Agent 能够在不同房间之间建立关联，产生更连贯、更自然的跨房间行为。

---

## 与 V3 的核心区别

V3 中，同名 Agent 在不同聊天室中是完全隔离的实例，各自只感知本房间的消息历史。

V3.1 中，Agent 的上下文扩展为**全局视角**：Agent 能看到自己参与的所有房间的消息历史，当前正在回复的房间作为主房间，其他房间的历史作为补充背景信息注入上下文。

| 能力 | V3 | V3.1 |
|------|-----|-------|
| 感知当前房间消息 | ✅ | ✅ |
| 感知其他房间消息 | ❌ | ✅ |
| 跨房间话题关联 | ❌ | ✅ |
| 房间间消息路由 | ✅ | ✅ |

---

## 功能特性

- Agent 发言时，上下文包含其参与的所有房间的完整消息历史
- Agent 可以在回复中主动引用或呼应其他房间发生的对话内容
- 各房间的发言规则不变：Agent 仍然只向当前房间发送消息，不会跨房间直接写入
- 调度器按**轮次交替推进**所有房间：每一大轮中，依次为每个房间调度一次发言，完成后进入下一大轮，如此循环直到所有房间达到各自的最大轮次。各房间在同一大轮内并发执行，互不等待
- 支持配置是否开启跨房间感知（可按 Agent 或全局开关控制）

## 效果演示

### 配置示例

两个聊天室并发运行：

- **general** 房间：alice、bob、charlie 聊日常话题
- **tech** 房间：bob、charlie 聊技术话题

Bob 和 Charlie 同时活跃于两个房间，能感知彼此在另一个房间的发言。

### 对话示例

```
=== tech 房间 ===
--- 第 1 轮 (bob @ tech) ---
bob: 最近在研究异步 IO，感觉并发模型越来越有意思。

--- 第 2 轮 (charlie @ tech) ---
charlie: 异步 IO 让我想起赫拉克利特——"万物皆流"，程序的执行也是如此。

=== general 房间 ===
--- 第 3 轮 (bob @ general) ---
bob: 刚才和 charlie 在 tech 房间聊到异步 IO，
     其实生活里也有类似的道理——很多事情不必等一件做完再开始下一件。

--- 第 4 轮 (charlie @ general) ---
charlie: Bob 说得对，我们在 tech 里聊的"万物皆流"，
         放到生活层面何尝不是如此——焦虑往往来自想要串行掌控一个并发的世界。
```

Bob 和 Charlie 在 general 房间的发言，自然地延续了他们在 tech 房间的讨论，无需任何手动触发。

## 验收标准

- [ ] Agent 的上下文中包含其参与的其他房间的消息历史
- [ ] Agent 能够在发言中自然引用其他房间的话题内容
- [ ] 跨房间上下文注入不影响消息路由，发言仍只写入当前房间
- [ ] 各房间对话轮次独立推进，并发调度行为与 V3 一致
- [ ] 不参与某房间的 Agent 看不到该房间的内容（权限边界正确）
- [ ] 支持通过配置开关控制是否启用跨房间感知
- [ ] 程序能正常退出

## 使用说明

### 配置文件

在 `config/agents_v3.1.json` 中新增字段：

```json
{
  "cross_room_context": true,
  "agents": [...],
  "chat_rooms": [...]
}
```

- `cross_room_context`：全局开关，设为 `true` 时所有 Agent 启用跨房间感知，设为 `false` 时行为与 V3 完全一致

### 运行

```bash
python src/main.py
```

### 输出

日志格式与 V3 保持一致，每条消息以房间名前缀标注，日志同时保存到 `logs/` 目录。
