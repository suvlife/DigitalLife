# V15: 房间工作中状态展示 - 产品文档

## 目标

优化房间中的运行态反馈，使用户在 Agent 处理房间任务时，能够明确看到“当前谁在工作”，降低长时间无消息时对系统状态的误判。

此前当 Agent 正在思考、推理或等待完成当前轮处理时，房间内可能长时间没有任何新消息，用户难以判断它是正在工作、暂时阻塞，还是已经失败。V15 之后，界面会提供明确的工作中状态提示，让用户知道当前轮到了谁、谁正在处理，以及失败后状态何时结束。

本文档描述的是**展示层反馈能力**，目标是让 Web Console 与 TUI 都能实时感知房间中的当前工作状态，而不改变现有消息数据模型。

---

## 功能特性

### 一、当前工作中可见

- **工作状态提示**：当 Agent 正在处理当前房间任务时，界面上会出现明确的“正在处理”状态提示。
- **当前成员可识别**：用户能知道当前轮到的是哪个 Agent，而不是只能等待下一条消息出现。
- **实时反馈**：用户无需等待最终发言落地，就能知道房间当前仍在推进中。
- **位置固定**：该状态显示在消息列表的最后一条之后，作为当前对话末尾的实时状态提示。

### 二、私聊与群聊都可见

- **私聊房间可见**：在 `PRIVATE` 房间中，用户可以明确看到当前 Agent 是否正在处理这轮对话。
- **群聊房间可见**：在 `GROUP` 房间中，用户也可以明确知道当前是哪位 Agent 正在处理本轮任务。
- **展示方式可按房间类型调整**：私聊和群聊可以使用不同的 UI 呈现，但都应提供一致的工作状态反馈。

### 三、失败状态可识别

- **失败可结束**：若当前处理中的 Agent 失败，界面中的工作中提示不应继续停留。
- **失败可感知**：用户应能看到当前处理已失败，避免误以为仍在正常运行。
- **状态切换及时**：成功、失败、取消后，展示状态都应及时更新。

### 四、不改现有消息体验

- **不改消息流逻辑**：V15 不引入消息排队、消息占位入库或消息原地替换。
- **不改房间写入规则**：现有消息何时入库、何时推送、何时可读的规则保持不变。
- **先解决可见性问题**：本版本优先补齐“看得到谁在工作”的反馈，不处理消息模型升级。

### 五、前端体验统一

- **Web 可见**：Web Console 能实时展示当前房间的工作状态。
- **TUI 可见**：TUI 前端也能看到一致的当前处理中提示。
- **体验一致**：不同前端对同一房间的工作状态保持一致理解。

---

## 用户价值

### 1. 降低“系统卡住了”的错觉

用户在等待 Agent 回复时，能够立刻看到“当前仍有人在处理”，而不是面对长时间静默的窗口。

### 2. 提升运行态透明度

用户可以明确知道当前轮到哪个 Agent，而不是只能靠猜测判断房间是否还在运行。

### 3. 降低阅读负担

本版本不改变现有消息流，只增加工作状态提示，因此能在不扰动阅读习惯的前提下提高可见性。

---

## 核心概念

### 工作中状态

界面上用于表达“当前房间中有 Agent 正在处理任务”的展示状态。

### 当前处理成员

当前轮到并正在处理房间任务的 Agent。私聊和群聊都需要向用户明确展示该信息。

## 产品边界

### V15 包含

- Agent 在私聊房间处理消息时展示“当前处理中”状态
- Agent 在群聊房间处理轮次任务时展示“当前谁在处理”
- Agent 失败、取消或结束时，工作状态能正确更新
- Web 与 TUI 的工作状态展示保持一致体验

### V15 不包含

- 私聊消息流中的占位消息
- 占位消息原地变为最终消息
- 私聊消息排队与可见性栅栏
- 逐字流式输入效果
- 复杂的已读/送达/回执体系

---

## 验收标准

### 房间体验

- 当 Agent 正在处理房间任务时，房间界面中能看到明确的工作状态提示
- `PRIVATE` 与 `GROUP` 房间都能识别当前正在处理的 Agent
- 工作状态显示在当前消息列表的最后一条之后
- 工作状态提示不会改变现有消息流顺序与写入行为

### 异常处理

- Agent 失败时，界面不会继续停留在“处理中”状态
- Agent 成功、失败、取消后，状态都能及时切换

### 多前端一致性

- Web Console 与 TUI 对同一房间的工作状态保持一致
